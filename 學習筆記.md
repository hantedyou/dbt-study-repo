# dbt + Snowflake 連線設定筆記

## 背景
使用 `dbt init --skip-profile-setup airbnb` 建立專案骨架後，手動設定 `airbnb/profiles.yml`
搭配 Snowflake 的 key-pair 認證方式連線，過程中遇到三個問題，記錄如下。

## 問題一：`private_key_passphrase` 型別錯誤

```
Runtime Error
  Credentials in profile "airbnb", target "dev" invalid: 1024 is not valid under any of the given schemas
```

**原因**：YAML 裡數字沒加引號會被解析成整數，但 dbt/Snowflake adapter 的 schema
要求 `private_key_passphrase` 必須是**字串**。

**修法**：加上引號

```yaml
private_key_passphrase: "1024"
```

## 問題二：`private_key` 應改為 `private_key_path`

原本寫法：

```yaml
private_key: /Users/hantedyou/.../rsa_key.p8
```

`private_key` 欄位語意上是「直接放金鑰內容字串」，若是要指向**檔案路徑**，應該用
`private_key_path` 欄位：

```yaml
private_key_path: /Users/hantedyou/.../rsa_key.p8
private_key_passphrase: "1024"
```

## 問題三：Snowflake `account` 格式錯誤

修好上面兩個問題後，`dbt debug` 的 profile 驗證通過，但連線測試失敗：

```
Database Error
  290404 (08001): 404 Not Found: post LY54946.snowflakecomputing.com:443/session/v1/login-request
```

**原因**：`account:` 欄位原本填的是 Snowflake **Account Name**（`LY54946`），
但 dbt 需要的是能組出正確登入網址的識別碼，格式通常是：

```
<account_locator>.<region>.<cloud_platform>
```

從 Snowsight「Admin → Accounts」頁面查到的實際資訊：

| 欄位 | 值 |
|---|---|
| Account Name | LY54946 |
| Locator | AG54694 |
| Cloud | AWS |
| Region | Asia Pacific (Seoul) → `ap-northeast-2` |

**修法**：改用 Locator + Region + Cloud

```yaml
account: AG54694.ap-northeast-2.aws
```

## 結果

```
Connection test: [OK connection ok]
```

## 心得 / 重點整理
- YAML 裡任何「看起來像數字，但其實要當字串用」的值，記得加引號。
- `private_key` vs `private_key_path`：前者放金鑰內容，後者放檔案路徑，別搞混。
- Snowflake 的 `account` 欄位不是「帳號名稱」，而是要能組出正確 API 網址的識別碼，
  最保險的做法是去 Snowsight 的 Admin → Accounts 頁面查 **Locator + Cloud + Region**。

## `dbt_project.yml` 裡 `+materialized` 的 `+` 是什麼

`dbt_project.yml` 的 `models:` 區塊是刻意設計成跟 `models/` 資料夾結構對應的巢狀 YAML：

```yaml
models:
  airbnb:
    +materialized: view   # 套用到 airbnb 專案下所有 model
    dim:
      +materialized: table  # 套用到 models/dim/ 底下所有 model
```

**問題**：同一層 YAML 裡，`dim` 這種 key 代表「往下一層資料夾」，`materialized` 這種
key 代表「要套用的設定屬性」，兩者長得都是普通 key，dbt 要怎麼分辨？

**舊做法（已棄用）**：靠猜——key 若符合已知設定名稱（`materialized`、`tags`、`enabled`...）
就當設定，否則當資料夾層級。容易產生歧義（例如剛好有個資料夾叫 `tags`）。

**現行做法**：設定屬性一律要加 `+` 前綴明確標示，沒加 `+` 的 key 一律視為資料夾層級。
沒加會出現 deprecation 警告：
```
[MissingPlusPrefixDeprecation] Missing '+' prefix on `airbnb`...
```

**`+` 是 dbt 專屬語法**，不是標準 YAML 語法。

**繼承規則**：config 由淺到深逐層覆寫（越靠近檔案的設定優先）。例如上面範例中，
`models/dim/` 底下的 model 會被覆寫成 `table`，其他資料夾的 model 仍沿用專案預設的 `view`。

## `ref()` 怎麼知道完整路徑（database.schema.object）

`{{ ref('src_listings') }}` 只需要給 model 名稱（對應檔名），完整路徑由 dbt 在編譯階段組出來：

1. **database / schema 預設值來自 `profiles.yml`** 的 target 設定（例如 `database: AIRBNB`、
   `schema: DEV`、`target: dev`）。
2. 若 `dbt_project.yml` 沒有針對該 model 設定 `+schema:`，也沒有自訂
   `generate_schema_name` macro，就直接用 target 的預設 schema。
3. 編譯後，`ref()` 會被替換成完整路徑（例如 `"AIRBNB"."DEV"."src_listings"`），
   純 SQL 才會真的送到 Snowflake 執行——可以用 `dbt compile` 產生的
   `target/compiled/.../*.sql` 檔案驗證編譯結果。

## `ephemeral` materialization 是什麼

`materialized` 常見三種：`view`、`table`、`ephemeral`。前兩者都會在資料庫裡建立
真實物件，`ephemeral` 則**完全不會**——不建 view、不建 table，Snowflake 裡完全看不到它。

**原理**：dbt 編譯時，把 ephemeral 模型的 SQL 整段**內嵌成 CTE**，塞進所有引用
它（`{{ ref(...) }}`）的下游模型的編譯結果裡，跟著下游模型一起執行一次。

適合用在單純做欄位篩選、改名等輕量中繼邏輯，不需要在資料庫留下實體物件的場合。
缺點：若同一個 ephemeral 模型被多個下游模型引用，它的 SQL 會被**重複內嵌、重複
執行**多次（不像 view/table 只需計算一次）。

| | 資料庫物件 | 執行時機 | 可被其他工具直接查詢 |
|---|---|---|---|
| Table | 實體表 | 建表時執行一次，資料落地 | 可以 |
| View | View 物件 | 每次查詢時執行 | 可以 |
| Ephemeral | 無 | 被 inline 成 CTE，隨下游模型一起執行 | 不行 |

## 用 `dbt compile` 驗證 ephemeral 如何被 inline

專案設定（`dbt_project.yml`）：

```yaml
models:
  airbnb:
    +materialized: view
    dim:
      +materialized: table
    src:
      +materialized: ephemeral
```

`models/src/src_hosts.sql`（原始）：

```sql
WITH raw_hosts AS (
    SELECT * FROM AIRBNB.RAW.raw_hosts
)
SELECT id AS host_id, NAME AS host_name, is_superhost, created_at, updated_at
FROM raw_hosts
```

`models/dim/dim_hosts_cleansed.sql`（原始）：

```sql
WITH src_hosts AS (
    SELECT * FROM {{ ref('src_hosts') }}
)
SELECT host_id, NVL(host_name, 'Anonymous') AS host_name, is_superhost, created_at, updated_at
FROM src_hosts
```

跑 `dbt compile` 後，`target/compiled/airbnb/models/dim/dim_hosts_cleansed.sql`：

```sql
WITH  __dbt__cte__src_hosts as (        -- ① src_hosts 整段邏輯被搬進來
WITH raw_hosts AS (
    SELECT * FROM AIRBNB.RAW.raw_hosts
)
SELECT
    id AS host_id, NAME AS host_name, is_superhost, created_at, updated_at
FROM raw_hosts
), src_hosts AS (                        -- ② dbt 自動補的轉接 CTE
    SELECT * FROM __dbt__cte__src_hosts
)
SELECT                                   -- ③ dim_hosts_cleansed.sql 原本的邏輯
    host_id,
    NVL(host_name, 'Anonymous') AS host_name,
    is_superhost, created_at, updated_at
FROM src_hosts
```

- ① `__dbt__cte__src_hosts`：ephemeral 模型的原始 SQL，dbt 自動改名成
  `__dbt__cte__<model_name>` 避免跟其他 CTE 撞名
- ② dbt 自動加一層 `src_hosts AS (SELECT * FROM __dbt__cte__src_hosts)`，讓
  下游模型裡的 `{{ ref('src_hosts') }}` 編譯後還能維持原本的名字可用
- ③ 才是 `dim_hosts_cleansed.sql` 自己寫的邏輯

最終只會 `CREATE TABLE dim_hosts_cleansed AS ...`（因為 `dim` 設定是 table），
Snowflake 上只看得到 `dim_hosts_cleansed`，`src_hosts` 完全不存在。

**心得**：`dbt compile` 只產生編譯後的 SQL、不會真的送去資料庫執行，是驗證
`ref()`、`is_incremental()`、ephemeral inline 等巢狀邏輯的最快方式——想確認
「dbt 到底組出什麼 SQL」時，直接看 `target/compiled/.../*.sql` 比用猜的準確。

## `dbt seed`：把 CSV 直接同步成 DB 裡的 table

`seeds/` 資料夾放的是**專案內建的靜態 CSV**（通常是不太會變動的參考資料，例如
`seed_full_moon_dates.csv`），跟 `models/` 底下「用 SQL 轉換資料」的邏輯不同，
seed 檔案本身就是資料本體。

執行：

```
dbt seed
```

dbt 會把 `seeds/` 底下每個 `.csv` 檔直接讀進來，在目標 schema 建立**同名的 table**
（例如 `seed_full_moon_dates.csv` → `AIRBNB.DEV.seed_full_moon_dates`），欄位型別
由 dbt 自動推斷 CSV 內容產生。之後就能像一般 model 一樣用 `{{ ref('seed_full_moon_dates') }}`
在別的 model 裡引用它。

**跟 `models/` 的差異：**

| | 資料來源 | 建立方式 |
|---|---|---|
| `models/*.sql` | 引用其他 model/source 用 SQL 轉換 | `dbt run` |
| `seeds/*.csv` | 專案內附的靜態 CSV 檔案本身 | `dbt seed` |

**適用場景**：資料量小、更新頻率低、適合直接放進版控的參考/對照資料（日期表、
國碼對照表、假日清單...），不適合拿來放正式的、量大或常變動的業務資料。

## `source()`：宣告「dbt 外部建立」的原始資料表

`_course_resources/course-resources.md` 第 23~88 行的 SQL（`CREATE OR REPLACE TABLE
raw_listings/raw_reviews/raw_hosts` + `COPY INTO ... FROM s3://...`）是**在 dbt 之外**、
直接用 Snowflake SQL 建出來的原始資料表，放在 `AIRBNB.RAW` schema。這些表不是 dbt
產生的，但下游的 `models/src/*.sql` 要引用它們。

`models/sources.yml` 的作用就是向 dbt **宣告這些外部建立的表存在**：

```yaml
sources:
  - name: airbnb        # source 群組名稱（邏輯命名空間，不一定等於 schema 名）
    schema: raw          # 實際 schema
    tables:
      - name: listings          # 邏輯名稱，model 裡用這個
        identifier: raw_listings # 實際 table 名稱

      - name: hosts
        identifier: raw_hosts

      - name: reviews
        identifier: raw_reviews
```

宣告後就能用 `{{ source('airbnb', 'hosts') }}` 引用，取代寫死路徑：

```diff
- AIRBNB.RAW.raw_hosts
+ {{ source('airbnb', 'hosts') }}
```

### Jinja 抽象層的概念

`{{ source(...) }}` 在編譯階段會被 dbt 展開成完整路徑（例如 `"AIRBNB"."RAW"."raw_hosts"`），
效果上跟寫死一樣，但多了一層**間接層**：

- model 的 SQL 不再直接耦合實體位置（database/schema/table 名稱），而是透過
  `sources.yml` 這個單一設定檔去查
- 之後 database 改名、schema 搬家、或换到不同環境，**只要改 `sources.yml` 一個地方**，
  不用一個一個去改每個引用到 raw 表的 model
- `identifier` 讓「邏輯名稱」跟「實際 table 名稱」可以不同（例如邏輯上叫 `listings`，
  實體叫 `raw_listings`），model 裡永遠用簡潔的邏輯名稱

**跟 `ref()` 的差異（機制相同、語意不同）：**
- `{{ ref('src_hosts') }}` → 引用**別的 dbt model**（dbt 建出來的東西）
- `{{ source('airbnb', 'hosts') }}` → 引用**dbt 外部建立的原始資料**（source）

兩者都會被 dbt 解析進 DAG，改用 `source()` 才能：
1. 在 `dbt docs generate` 的 lineage graph 裡把 raw 表畫成 DAG 的起點
2. 支援 `dbt source freshness` 檢查資料新鮮度
3. 用 `dbt test -s source:airbnb.listings` 單獨測試某個 source
4. 對 source 表本身加 `data_tests`（例如 not_null、unique）

如果繼續寫死 `AIRBNB.RAW.raw_hosts`，dbt 完全不知道這張表的存在，這些功能全部用不了。

## `dbt source freshness`：檢查 source 資料新不新鮮

在 `sources.yml` 的 table 底下加 `config`，才能讓 dbt 知道怎麼判斷這張 source 表的資料夠不夠新：

```yaml
- name: reviews
  identifier: raw_reviews
  config:
    loaded_at_field: date          # 用哪個欄位當「資料時間」
    freshness:
      warn_after: {count: 1, period: hour}   # 超過 1 小時沒更新 → WARN
      error_after: {count: 24, period: hour} # 超過 24 小時沒更新 → ERROR
```

執行：

```
dbt source freshness
```

**原理**：dbt 對每個有設定 `freshness` 的 source table 執行
`SELECT MAX(loaded_at_field) FROM <source table>`，拿這個最大值跟「現在時間」比較：
- 差距 < `warn_after` → PASS
- 差距 >= `warn_after` 但 < `error_after` → WARN
- 差距 >= `error_after` → ERROR

**用途**：確認上游（外部灌進 `RAW` schema 的資料）有沒有太久沒更新，避免下游 model
一直在算「過期資料」卻沒人發現。`loaded_at_field` 沒設定的 source table 不會被檢查
（dbt 不知道要看哪個欄位當時間）。

## 用 `echo $?` 看上一個指令的 exit code

Unix shell 慣例：指令執行完會回傳一個數字當作 exit code，`0` 代表成功，非 0 代表失敗
（不同工具的非 0 數字可能代表不同的失敗類型）。`$?` 是 shell 內建變數，永遠存著
**上一個指令**的 exit code：

```
dbt source freshness
echo $?
```

如果 `dbt source freshness` 全部 PASS，`echo $?` 印出 `0`；只要有任何一個 source
是 ERROR（WARN 不確定，需要實測確認是否也會讓 exit code 非 0），`echo $?` 就會印出非 0。

**用途**：CI/CD pipeline 裡常靠 exit code 判斷要不要繼續往下跑（例如 freshness
失敗就擋下後續的 `dbt run`），`echo $?` 是本機手動驗證這個機制最快的方式。

## `dbt snapshot` 的 `timestamp` strategy 只認 `updated_at`，不比對其他欄位

`snapshots/raw_hosts_snapshot.yml`：

```yaml
snapshots:
  - name: scd_raw_hosts
    relation: source('airbnb', 'hosts')
    config:
      unique_key: id
      strategy: timestamp
      updated_at: updated_at
      hard_deletes: invalidate
```

**踩雷經過**：直接對 `raw_hosts` 下 SQL 改資料

```sql
UPDATE AIRBNB.RAW.raw_hosts
SET is_superhost = 'f'
WHERE id = 1581;
```

跑完 `dbt snapshot` 完全沒反應，`scd_raw_hosts` 沒有新增任何一筆 row。

**原因**：`strategy: timestamp` 判斷「這筆資料有沒有變化」的唯一依據是
`updated_at` 欄位的值有沒有變大，**不會**去逐欄比對 `is_superhost` 這種一般欄位
內容是否不同。上面的 UPDATE 只改了 `is_superhost`，沒有一併更新 `updated_at`，
對 dbt 來說「時間戳沒變 = 沒有變化」，snapshot 自然不會記錄新版本。

**修法**：UPDATE 時要一併更新 `updated_at`：

```sql
UPDATE AIRBNB.RAW.raw_hosts
SET is_superhost = 'f', updated_at = CURRENT_TIMESTAMP()
WHERE id = 1581;
```

**心得**：`timestamp` strategy 依賴上游系統老實維護 `updated_at`，如果上游更新
資料時漏更新這個欄位，snapshot 會悄悄地「偵測不到變化」且不會報錯，很難察覺。
另一種 `strategy: check` 則是逐欄比對內容（可用 `check_cols: all` 或指定欄位清單），
不依賴時間戳欄位是否被正確維護，較適合不確定上游 `updated_at` 是否可靠的情境。

## `models/schema.yml` 的檔名不是 dbt 規定的，`schema.yml` 只是慣例

**問題**：`data_tests`（如 `unique`、`not_null`）都寫在 `models/schema.yml` 裡，但
`schema.yml` 這個檔名沒有在任何地方被明確指定過，dbt 是怎麼知道要讀這個檔案的？

**原理**：dbt 找設定檔靠的不是檔名，而是「掃描 + 內容結構」：

1. dbt 會掃描 `dbt_project.yml` 裡 `model-paths` 設定的資料夾（預設 `models/`），
   把底下**所有** `.yml` / `.yaml` 檔案都讀進來——不管檔名叫什麼都會被讀到。
2. 讀進來後，dbt 看的是檔案內容的**頂層 key**（`models:`、`sources:`、`seeds:`、
   `snapshots:`... 這些是 dbt 認得的保留字），只要有符合的 key，內容就會被解析。
3. `models:` 底下每個項目靠 `name:` 欄位去匹配實際的 model 檔案（例如
   `name: dim_listings_cleansed` 對應 `models/dim/dim_listings_cleansed.sql`），
   跟這個 yml 檔案本身叫什麼名字完全無關。

**心得**：`schema.yml` 只是社群慣例（多數教學、範例都這樣取名），不是 dbt 規定的
檔名。可以把測試設定拆成多個 yml（例如一個 model 一個 `_dim_listings.yml`），也可
以全部塞在一個大檔案裡，效果一樣。這跟 Python `import` 那種「檔名/路徑本身就是
識別碼」的機制不同，dbt 的 yml 比較像是丟進同一個池子，靠內容裡的頂層 key 和
`name:` 欄位自己表明身份，不靠檔名。

## dbt 內建的四種 generic tests

dbt 開箱即用（不用額外安裝套件）就有四種通用測試，寫在 `data_tests:` 底下，
套用在 `models:` 裡某個 model 的某個欄位上：

| 測試 | 用途 |
|---|---|
| `unique` | 該欄位的值不可重複 |
| `not_null` | 該欄位不可為 NULL |
| `accepted_values` | 該欄位的值必須落在指定的清單內 |
| `relationships` | 該欄位的值必須存在於另一張 table 的指定欄位（外鍵完整性） |

**範例**（`models/schema.yml`）：

**原理**：這四種測試本質上都是 dbt 內建的 **SQL 樣板（macro）**，執行 `dbt test`
時，dbt 會把設定值代入樣板、組成一段「找出違規 row」的 SELECT 查詢去 Snowflake
執行——查出來 row 數為 0 代表 PASS，>0 代表有資料違反規則、測試 FAIL。可以用
`dbt compile` 產生的 `target/compiled/.../*.sql` 驗證每個測試實際組出的查詢邏輯，
跟之前驗證 `ref()` / ephemeral inline 的方式一樣。

**心得**：這四種是最基礎的資料品質檢查（唯一性、必填、枚舉值、參照完整性），
`unique` + `not_null` 通常用在 primary key，`relationships` 則是拿來驗證兩張表
之間的關聯是否乾淨（例如 fact table 的外鍵是否都能在 dim table 找到對應資料）。

## `dbt test`：執行測試，`-x` / `--fail-fast` 讓它一遇到錯誤就停

```
dbt test
```

會把 yml 裡設定的所有 `data_tests`（`unique`、`not_null`、`accepted_values`、
`relationships`...）全部組成 SQL 查詢，逐一送到 Snowflake 執行，預設會**跑完全部**
測試才回報結果，就算中途有測試 FAIL 也會繼續跑下一個。

```
dbt test -x
dbt test --fail-fast
```

`-x` 是 `--fail-fast` 的簡寫，加上這個 flag 後，只要**第一個**測試 FAIL，dbt 就會
立刻中止、不再繼續執行剩下的測試。

**用途**：平常開發階段想看到「全部有哪些測試沒過」，用預設的 `dbt test` 比較好
（一次看到所有問題）。但在 CI/CD pipeline 或資料量很大、測試很花時間的情境，
用 `-x` 可以在發現第一個問題時就立刻停下、盡快讓後續流程（例如擋下 `dbt run`）
知道有問題，不用浪費時間跑完剩下的測試。

## `+store_failures: true`：把測試失敗的資料存下來，不只是回報 PASS/FAIL

`dbt_project.yml` 裡對 `data_tests:` 這個頂層設定加上：

```yaml
data_tests:
  +store_failures: true
```

**行為**：預設 `dbt test` 只會告訴你「這個測試 PASS 還是 FAIL、違規幾筆」，看不到
是**哪幾筆資料**違規。加上 `store_failures: true` 後，只要有測試 FAIL，dbt 會把
違規的那些 row **實際建成一張 table**，存到 `DEV_DBT_TEST__AUDIT` 這個 schema
底下（schema 名稱通常是 `<target_schema>_dbt_test__audit`，這裡 target 是
`DEV`，所以是 `DEV_dbt_test__audit`）。

`dbt test` 跑完後，FAIL 的測試會在 console 印出可以直接查詢的位置，例如：

```
See test failures:
  --------------------------------------------------------------------------------------------------------
  select * from AIRBNB.DEV_dbt_test__audit.accepted_values_dim_listings_c_2a86f637e70df013556a8a127cb46aa1
  --------------------------------------------------------------------------------------------------------
```

**table 命名細節**：不是單純的 `accepted_values_dim_listings_cleansed_room_type`
這種直覺全名——Snowflake 物件識別碼有長度限制，dbt 對過長的測試名稱會做
**截斷 + hash** 處理：前段保留可辨識的名稱（`accepted_values_dim_listings_c`），
後面接一段依測試完整定義（model、欄位、參數）算出來的 32 位 hash
（`2a86f637e70df013556a8a127cb46aa1`），確保就算截斷後前段撞名，整個 table 名稱
還是唯一。這串 SQL 是 dbt 貼心印出來的，複製貼上就能直接查是哪幾筆資料違規，
不用自己去 `DEV_dbt_test__audit` 底下用眼睛找。

**心得**：這個設定跟之前學的 `+materialized`、`+schema` 一樣是 dbt 專屬的 `+`
前綴 config 語法，套用範圍是「整個專案所有 data_tests」。放在 `dbt_project.yml`
是全域套用，也可以像 model 一樣改成只對單一測試加（在 yml 的 `data_tests:` 底下
用 `config: {store_failures: true}`）。實務上很有用：測試 FAIL 時不用自己重寫
SQL 去反查是哪幾筆資料出問題，直接複製 console 印出的 SQL 查詢就好。

## 用 `+schema` 自訂 test failure 存放的 schema 名稱

`data_tests` 底下再加一行（見 `dbt_project.yml`）：

```yaml
data_tests:
  +store_failures: true
  +schema: _test_failures
```

**行為**：`+schema` 不是「直接取代」目標 schema，而是跟 model 的 `+schema` 邏輯
一樣，套用 dbt 內建的 `generate_schema_name` macro（專案裡沒有自訂這個 macro，
所以走預設規則）：**實際 schema = `<target_schema>_<自訂 schema>`**。這裡
target 是 `DEV`，所以最終會建在 `DEV__test_failures`（`DEV` + `_` + 
`_test_failures`，中間變成兩個底線），而不是原本預設的 `DEV_dbt_test__audit`。

**心得**：跟 model 的 `+schema` 是同一套機制、同一個 macro，只是套用對象換成
`data_tests`。想要「完全自訂、不要被加上 target_schema 前綴」的話，需要在
`macros/` 底下覆寫 `generate_schema_name`（例如判斷 custom schema 是
`_test_failures` 時直接回傳，不 concat target schema），這個專案目前沒有這樣
的自訂 macro，所以還是會保留 `DEV_` 前綴。

## `dbt test -x` 遇到 FAIL 後：單執行緒下依然可重現的 CANCEL/PASS 現象

故意在 `models/schema.yml` 把 `room_type` 的 `accepted_values` 清單打錯字
（`'Hotel room break'` 而非實際資料的 `'Hotel room'`）製造一筆真的會 FAIL 的測試，
拿來驗證 `-x` 的行為，重跑兩次結果一致：

```
1 of 9 START test accepted_values_dim_hosts_cleansed_is_superhost__t__f ........ [RUN]
1 of 9 PASS  accepted_values_dim_hosts_cleansed_is_superhost__t__f .............. [PASS in 1.64s]
2 of 9 START test accepted_values_dim_listings_cleansed_room_type__...__break .. [RUN]
2 of 9 FAIL 1 accepted_values_dim_listings_cleansed_room_type__...__break ...... [FAIL 1 in 1.08s]
3 of 9 START test not_null_dim_hosts_cleansed_host_id .......................... [RUN]
CANCEL query test.airbnb.not_null_dim_hosts_cleansed_host_id.de84c3da05 ........ [CANCEL]
3 of 9 PASS  not_null_dim_hosts_cleansed_host_id ................................ [PASS in 1.25s]

Completed with 8 errors, 0 partial successes, and 0 warnings:
Done. PASS=1 WARN=0 ERROR=1 SKIP=7 NO-OP=0 TOTAL=9
```

**現象**：跑的時候 `Concurrency: 1 threads`，是單執行緒依序執行，所以這不是隨機
race condition，是每次都會重現的固定行為：

1. test 2（`accepted_values`）FAIL，觸發 fail-fast
2. test 3 的查詢已經送出去了，dbt 對它送出 `CANCEL`，但查詢在 cancel 生效前就已
   經跑完並回傳結果，log 因此還是印出 `3 of 9 PASS`
3. test 4~9 完全沒有 `START`，直接被跳過（不會顯示在 log 裡）
4. 但**最終統計** `PASS=1`，並沒有把 test 3 算進去——代表 dbt 在排程層級已經把
   test 3 標記為「這次 run 被中止範圍內的節點」，跟這個查詢本身有沒有實際拿到
   PASS 結果是分開判斷的兩件事，最終統計採用的是排程狀態，不是查詢結果
5. `Completed with 8 errors` 的 `8` = `1 個真的 FAIL + 7 個 SKIP` 加總，這句話把
   兩種情況都算進「需要關注的問題」，不是真的有 8 個測試各自失敗

**心得**：`-x` / `--fail-fast` 觸發後，「已經送出去的查詢」跟「dbt 排程要不要繼續」
是兩條分開的邏輯，即使某個 in-flight 查詢最後順利回傳 PASS，只要它是在 fail-fast
觸發之後才完成的，dbt 仍然會把它計入中止範圍、不算進最終的 PASS 計數。看 log 時
不能只看單一測試那行印出的 PASS/FAIL，要以最後的 `Done. PASS=... ERROR=... SKIP=...`
統計為準。

## singular test：直接寫一個 SQL 檔案當測試

`tests/` 資料夾放的是「找出違規資料」的 SQL 查詢（見 `tests/dim_listings_minimum_nights.sql`，
檢查 `minimum_nights` 不能小於 1）。跟目前學過的四種 generic test（`unique`、
`not_null`、`accepted_values`、`relationships`）不同：generic test 是可套用在
多個欄位/model 上的**通用樣板（macro）**，singular test 則是**一次性、針對特定
商業邏輯手寫的 SQL**，沒有樣板、想測什麼就直接寫查詢。

**判斷邏輯**：檔名就是測試名稱（`dim_listings_minimum_nights`）。這段 SQL 查出來
的 row 數 > 0 → FAIL（查到的都是違規資料），0 筆 → PASS。本質上跟 generic test
是同一套判斷邏輯（找違規 row、數量決定 PASS/FAIL），只是 generic test 靠 macro
產生 SQL，singular test 直接手寫。

**心得**：`dbt test` 執行時，singular test 會跟其他 generic test 排在同一個
執行序列裡，`store_failures`、`-x` 這些機制對它一樣適用，沒有特殊待遇。

## `dbt test -s`：只跑指定的測試，不用每次全跑

```
dbt test -s dim_listings_minimum_nights
```

`-s`（`--select`）是 dbt 的 node 選擇語法，`dbt run -s` 可以只跑指定 model，
同樣的語法用在 `dbt test` 上就能只跑指定的測試（不管是 singular test 還是
generic test 產生出來的測試 node）。

也可以直接指定 model 名稱，讓 dbt 抓出所有跟這個 model 有關的測試一起執行：

```
dbt test -s dim_listings_cleansed
```

**心得**：dbt 預設的「indirect selection」行為是只要一個測試有引用到被選中的
model（不管是 schema.yml 裡的 generic test，還是 tests/ 資料夾的 singular test），
就會一併被選進來執行。開發階段改了某個 model 只想驗證相關測試時，不用每次
`dbt test` 全部 9 個以上都重跑一遍，節省時間。

## unit test：測「SQL 邏輯」，不是測「真實資料」

設定見 `airbnb/models/mart/unit_tests.yml`。跟前面學的 generic test / singular
test 有本質差異：

| | 測試對象 | 資料來源 |
|---|---|---|
| generic / singular test | 資料庫裡**已經跑出來的實際資料**有沒有問題 | 真實 model 的查詢結果 |
| unit test | model **SQL 邏輯本身**對不對 | 手寫的假 input，不碰真實資料 |

`unit_tests.yml` 裡的 `given` 是幫每個 `input`（可以是 `ref()` 的 model，也可以
是別的 model）餵幾筆手造的 rows，`expect` 是預期這個 model 邏輯跑完後應該產出
的結果，兩者不一致就是 FAIL。

執行方式：

```
dbt test -s unittest_fullmoon_matcher
dbt test -s mart_fullmoon_reviews
```

**心得**：`-s` 後面可以接 unit test 自己的 `name`（`unittest_fullmoon_matcher`），
也可以直接接它所屬的 `model`（`mart_fullmoon_reviews`）——跟前面 `dbt test -s`
用 model 名稱抓出所有相關測試是同一套 indirect selection 機制，unit test 也適用。
好處是不用連真實資料表都建好、跑完整個 pipeline，就能單獨驗證這段 SQL 邏輯
（例如滿月判斷）寫得對不對，尤其適合邊界條件（前一天、當天、後一天）這種
不容易剛好在真實資料裡湊齊的案例。

## 自定義 generic test：`tests/generic/` 放的是自己寫的測試樣板

內建的 `unique`、`not_null`、`accepted_values`、`relationships` 之外，也可以自己
寫一個 generic test 樣板，供多個 model/欄位共用。範例見
`airbnb/tests/generic/positive_values.sql`：

```sql
{% test positive_values(model, column_name) %}
select * from {{ model }} where {{ column_name }} <= 0
{% endtest %}
```

**放置位置**：`{% test %}` 巢狀 macro 放在 `tests/generic/` 資料夾底下（也可以放
`macros/` 底下，dbt 認的是 `{% test %}` 這個巢狀語法本身，不是資料夾路徑；
`tests/generic/` 只是慣例，跟前面學過「`schema.yml` 檔名不是規定、內容結構才是
關鍵」是同一種概念）。

**命名規則**：`{% test positive_values(...) %}` 裡的 `positive_values` 就是這個
generic test 的名字，`schema.yml` 裡引用時必須完全對上：

```yaml
- name: minimum_nights
  data_tests:
    - positive_values
```

（見 `airbnb/models/schema.yml` 裡 `dim_listings_cleansed` 的 `minimum_nights` 欄位）

**踩雷經驗**：一開始 `dbt test -s dim_listings_cleansed` 報錯，原因是
`schema.yml` 裡寫的測試名稱跟 `positive_values.sql` 裡 `{% test %}` 定義的名字
沒對上，dbt 找不到對應的 macro。

**心得**：自訂 generic test 本質上跟 dbt 內建的四種 generic test 是同一套機制
（macro 樣板 + 代入參數組成 SQL），差別只在內建的樣板 dbt 自帶、自訂的要自己寫。
`model` 跟 `column_name` 是 dbt 呼叫 generic test 時固定會傳入的參數，`model`
是被測試的 model 物件、`column_name` 是套用測試的欄位名稱，測試邏輯裡直接用
`{{ column_name }} <= 0` 這種條件找違規 row，跟內建測試的判斷方式（查到 row
就是 FAIL）一致。

## 自訂 generic test 可以帶自己的參數，呼叫時用 `arguments:` 指定

`model`（跟 column 層級的 `column_name`）是 dbt 固定會傳入的參數，除此之外還可以
自己在 `{% test %}` 定義裡加任意參數，呼叫時透過 `arguments:` 指定值。範例見
`airbnb/tests/generic/minimum_row_counts.sql`：

```sql
{% test minimum_row_count(model, min_row_count) %}
select count(*) as cnt from {{ model }}
having cnt < {{ min_row_count }}
{% endtest %}
```

`min_row_count` 是自訂參數（不是 dbt 固定傳的），套用在 `schema.yml`（model 層級，
不是 column 層級，因為這個測試檢查的是整張表的 row 數，不是特定欄位）：

```yaml
models:
  - name: dim_listings_cleansed
    data_tests:
      - minimum_row_counts:
          arguments:
            min_row_count: 1000
```

**踩雷經驗**：這裡實際發生過名稱對不上的問題——macro 定義的名字是**單數**
`minimum_row_count`，但 `schema.yml` 裡寫成**複數** `minimum_row_counts`，
兩邊沒對齊導致 `dbt test -s dim_listings_cleansed` 找不到這個 test。跟前面
`positive_values` 那次一樣，dbt 只認 `{% test %}` 裡定義的名字，`.sql` 檔名
（`minimum_row_counts.sql`）叫什麼完全不影響、也不會被拿來比對。

**心得**：自訂 generic test 的參數命名沒有限制，`{% test %}` 括號裡除了
`model`/`column_name` 之外的每個參數，呼叫時都要在 `arguments:` 底下用同名的
key 指定值，等於是幫這個測試樣板開一個自訂的「設定介面」。
