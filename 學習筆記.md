# dbt + Snowflake 連線設定筆記

## 背景
使用 `dbt init --skip-profile-setup airbnb` 建立專案骨架後，手動設定 `airbnb/profiles.yml`
搭配 Snowflake 的 key-pair 認證方式連線，過程中遇到三個問題，記錄如下。

## 問題一：`private_key_passphrase` 型別錯誤

```
Runtime Error
  Credentials in profile "airbnb", target "dev" invalid: 1024 is not valid under any of the given schemas
```

**原因**：YAML 裡數字沒加引號會被解析成整數，但 dbt/Snowflake adapter 的 schema
要求 `private_key_passphrase` 必須是**字串**。

**修法**：加上引號

```yaml
private_key_passphrase: "1024"
```

## 問題二：`private_key` 應改為 `private_key_path`

原本寫法：

```yaml
private_key: /Users/hantedyou/.../rsa_key.p8
```

`private_key` 欄位語意上是「直接放金鑰內容字串」，若是要指向**檔案路徑**，應該用
`private_key_path` 欄位：

```yaml
private_key_path: /Users/hantedyou/.../rsa_key.p8
private_key_passphrase: "1024"
```

## 問題三：Snowflake `account` 格式錯誤

修好上面兩個問題後，`dbt debug` 的 profile 驗證通過，但連線測試失敗：

```
Database Error
  290404 (08001): 404 Not Found: post LY54946.snowflakecomputing.com:443/session/v1/login-request
```

**原因**：`account:` 欄位原本填的是 Snowflake **Account Name**（`LY54946`），
但 dbt 需要的是能組出正確登入網址的識別碼，格式通常是：

```
<account_locator>.<region>.<cloud_platform>
```

從 Snowsight「Admin → Accounts」頁面查到的實際資訊：

| 欄位 | 值 |
|---|---|
| Account Name | LY54946 |
| Locator | AG54694 |
| Cloud | AWS |
| Region | Asia Pacific (Seoul) → `ap-northeast-2` |

**修法**：改用 Locator + Region + Cloud

```yaml
account: AG54694.ap-northeast-2.aws
```

## 結果

```
Connection test: [OK connection ok]
```

## 心得 / 重點整理
- YAML 裡任何「看起來像數字，但其實要當字串用」的值，記得加引號。
- `private_key` vs `private_key_path`：前者放金鑰內容，後者放檔案路徑，別搞混。
- Snowflake 的 `account` 欄位不是「帳號名稱」，而是要能組出正確 API 網址的識別碼，
  最保險的做法是去 Snowsight 的 Admin → Accounts 頁面查 **Locator + Cloud + Region**。

## `dbt_project.yml` 裡 `+materialized` 的 `+` 是什麼

`dbt_project.yml` 的 `models:` 區塊是刻意設計成跟 `models/` 資料夾結構對應的巢狀 YAML：

```yaml
models:
  airbnb:
    +materialized: view   # 套用到 airbnb 專案下所有 model
    dim:
      +materialized: table  # 套用到 models/dim/ 底下所有 model
```

**問題**：同一層 YAML 裡，`dim` 這種 key 代表「往下一層資料夾」，`materialized` 這種
key 代表「要套用的設定屬性」，兩者長得都是普通 key，dbt 要怎麼分辨？

**舊做法（已棄用）**：靠猜——key 若符合已知設定名稱（`materialized`、`tags`、`enabled`...）
就當設定，否則當資料夾層級。容易產生歧義（例如剛好有個資料夾叫 `tags`）。

**現行做法**：設定屬性一律要加 `+` 前綴明確標示，沒加 `+` 的 key 一律視為資料夾層級。
沒加會出現 deprecation 警告：
```
[MissingPlusPrefixDeprecation] Missing '+' prefix on `airbnb`...
```

**`+` 是 dbt 專屬語法**，不是標準 YAML 語法。

**繼承規則**：config 由淺到深逐層覆寫（越靠近檔案的設定優先）。例如上面範例中，
`models/dim/` 底下的 model 會被覆寫成 `table`，其他資料夾的 model 仍沿用專案預設的 `view`。

## `ref()` 怎麼知道完整路徑（database.schema.object）

`{{ ref('src_listings') }}` 只需要給 model 名稱（對應檔名），完整路徑由 dbt 在編譯階段組出來：

1. **database / schema 預設值來自 `profiles.yml`** 的 target 設定（例如 `database: AIRBNB`、
   `schema: DEV`、`target: dev`）。
2. 若 `dbt_project.yml` 沒有針對該 model 設定 `+schema:`，也沒有自訂
   `generate_schema_name` macro，就直接用 target 的預設 schema。
3. 編譯後，`ref()` 會被替換成完整路徑（例如 `"AIRBNB"."DEV"."src_listings"`），
   純 SQL 才會真的送到 Snowflake 執行——可以用 `dbt compile` 產生的
   `target/compiled/.../*.sql` 檔案驗證編譯結果。
